## 重要进展：成功配置Claude Code使用Ollama本地模型

### 问题背景
- 用户尝试让OpenClaw助理与Claude Code交互，但Claude Code始终显示"Not logged in"错误
- 用户已在Windows中安装Claude Code并配置了Ollama本地模型
- OpenClaw助理运行在WSL2环境中，无法直接访问Windows中的Claude Code认证状态

### 解决方案探索过程
1. **初始尝试**：使用Claude Code的`--permission-mode bypassPermissions --print`参数，但认证失败
2. **检查配置**：发现Claude Code配置文件使用OpenRouter API，但API密钥可能无效或需要登录
3. **环境变量方案**：用户建议将环境变量写入OpenClaw服务配置文件
4. **Ollama发现**：确认WSL可以访问Windows中的Ollama服务，可用模型包括：
   - qwen2.5:14b
   - bge-m3:latest
   - qwen-7b-q4:latest
   - qwen-vl-7b:latest
   - deepseek-7b:latest
   - gemma-3-4b:latest
   - modelscope.cn/qwen/Qwen2.5-7B-Instruct-GGUF:latest

### 最终解决方案
通过以下配置成功让Claude Code使用Ollama本地模型：

1. **环境变量配置**（`~/.config/openclaw/env`）：
   ```
   ANTHROPIC_AUTH_TOKEN=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
   ANTHROPIC_BASE_URL=http://host.docker.internal:11434
   CLAUDE_CODE_NO_UPDATE_CHECK=1
   CLAUDE_CODE_DISABLE_TELEMETRY=1
   ```

2. **Claude Code启动参数**：
   ```
   claude --bare --settings '{"provider": "ollama", "model": "qwen2.5:14b"}'
   ```

### 关键发现
- `ANTHROPIC_BASE_URL`可以指向Ollama服务地址（`http://host.docker.internal:11434`）
- `--bare`参数启用最小化模式，跳过认证检查
- `--settings`参数可以指定provider和model，覆盖默认配置
- WSL通过`host.docker.internal`可以访问Windows主机的服务

### 测试验证
成功运行测试命令：
- `claude --bare --settings '{"provider": "ollama", "model": "qwen2.5:14b"}' "写一个Python hello world"`
- Claude Code正确响应，生成Python代码

### 技术要点
1. **跨平台访问**：WSL通过特殊主机名访问Windows服务
2. **环境变量优先级**：OpenClaw网关服务的环境变量配置优先于用户shell环境
3. **Provider配置**：Claude Code支持通过`--settings`动态指定provider和model
4. **最小化模式**：`--bare`参数跳过认证和更新检查，适合自动化场景

### 经验教训
- 当遇到认证问题时，考虑使用`--bare`模式绕过
- Ollama可以作为Claude Code的后端provider
- 环境变量配置是解决跨环境认证问题的有效方法
- WSL与Windows服务互通的特殊主机名：`host.docker.internal`
